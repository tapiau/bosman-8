# Analysis
This folder looks like a Z80 ROM dump and disassembly for the Bosman-8 "Uczen-2"
Polish educational microcomputer. The ROM is a 32KB MBM27256 EPROM mapped at
`0x0000-0x7FFF`. The system is CP/M-compatible and explicitly prints:
`"Dyskowy system operacyjny  CPM/R v2.5    TPA=60KB"` (a Polish CP/M variant).

The disassembly strongly matches a classic CP/M layout:
- low-memory jump vectors at `0x0000`, `0x0005`, `0x0038`
- BIOS at low ROM
- BDOS entry at `0x0005`
- CCP (command processor) and commands in ROM
- a ROM-resident text editor
- hardware set: Z80 + WD1793 FDC + Z80 PIO + Z80 CTC + MC6845 CRT

Origin clues:
- `rom.asm` contains a `z80dasm 1.1.6` header with the exact command line used.
  That makes it a raw disassembly of `rom.bin`.
- The `01_..` and `03_..` files include extensive human commentary and
  structured "MODULE" headers, suggesting a manual split/annotation of
  `rom.asm` into themed chunks. The rest appear to be mostly raw disassembly.

File-by-file notes (best-effort):
- `01_vectors_and_boot_data.asm`
  - Reset vectors, warm/cold boot patches, RST handlers, NMI, boot banners,
    error strings like `RAMDYSK uszkodzony` and `ROM uszkodzony`.
  - Contains CRT 6845 parameter tables and system config data.
  - Commentary explicitly names Bosman-8 Uczen-2 and CPM/R v2.5.
- `02_hardware_init.asm`
  - Cold boot flow: ROM checksum (ports `0x06/0x07`), stack setup, BIOS jump
    table copy to RAM at `0xF200`, IM2 setup, CTC/PIO/CRT init, DIP switch read
    at `0x98` for configuration, RAM clear, disk config probing.
- `03_bios.asm`
  - Core CP/M BIOS: `SELDSK`, `READ`, `WRITE`, console I/O, reader, printer
    status, disk geometry handling, WD1793 FDC access via RAM-based stubs.
- `04_bios_tables.asm`
  - BIOS jump table (ROM copy for RAM vectors), disk parameter tables,
    translation/data tables, and helper stubs. Mostly data/jump entries.
- `05_bdos.asm`
  - BDOS dispatcher and function implementations. Typical CP/M-style
    function switch on `C` and stack switching for BDOS reentrancy.
- `06_ccp_and_commands.asm`
  - Console Command Processor and built-in commands. Input line editing
    (Backspace, DEL, Ctrl-U, etc.), command parsing, error strings like
    `NIE MA TAKIEJ KOMENDY` ("no such command").
- `07_hw_drivers.asm`
  - Low-level hardware routines and helpers; likely keyboard, display,
    printer, disk access utilities used by BIOS/CCP/editor.
- `08_text_editor.asm`
  - ROM-resident text editor starting around `0x5B00`. Uses BDOS calls (`0005`)
    and screen/keyboard routines. Looks like a simple line-oriented editor.
- `rom.asm`
  - Full raw disassembly of `rom.bin` generated by `z80dasm`.
- `rom.bin`
  - The original 32KB ROM image.

## Networking Notes
I did not find explicit network protocol or multi-node classroom "server/client"
features in this ROM. What does appear is a serial I/O path:
- BIOS `CONOUT` can route output to a serial port via I/O port `0x98`.
- `CONST`/input routines poll the same port for serial status.
- DIP switches are also read from `0x98`, suggesting a shared status/data port.

That points to a basic serial console or printer/aux device, not a network
stack or classroom bus. If this is a "client" model, the ROM still looks like a
standard CP/M-compatible system with optional serial I/O, not a networked
client with explicit teacher-server protocol logic.

Best-guess (no hardware access):
- The RS-485 multidrop bus likely lives on an external communication board
  (per the DS75176AN evidence). The ROM probably only knows about a serial
  data port and perhaps a direction/enable latch bit, which would be handled
  in hardware or via a small I/O routine. I see no obvious protocol framing,
  node addressing, or polling logic in the ROM strings or BIOS/CCP code.
- That implies the "client" vs "server" distinction may be hardware-only or
  disk/utility-based, not a different ROM image.

## Hardware Photo Notes (From `hardware-description.txt`)
There is a separate hardware analysis noting two DS75176AN RS-485/RS-422
transceivers on a related board, plus open-collector 7406 drivers and three
analog adjustment sections with trimmers. That strongly suggests an RS-485/
RS-422 multidrop physical layer, likely with external direction control and
signal conditioning. This would still appear in ROM as basic serial I/O plus
direction/enable control bits, not as a distinct network stack.
